# First Line of Code (3rd Ed.) · Chapter 13 — Jetpack Architecture Components

## Chapter 13: Jetpack

Jetpack is a collection of development components designed to help us write cleaner code and simplify development. Many components are tailored for the MVVM architecture.

### 13.2 ViewModel

- Purpose: helps Activities share some of the workload, specifically for storing UI-related data.

- Data isn't lost on rotation; only destroyed when the Activity is destroyed

```
implementation 'androidx.lifecycle:lifecycle-extensions:2.2.0'
```

- Typically create one ViewModel per Activity and Fragment

- Counter example

1. Create ViewModel

```
class MainViewModel : ViewModel(){

    var counter = 0
}
```

1. Layout button click, display data

2. Logic code

```
class MainActivity : AppCompatActivity() {

    lateinit var viewModel: MainViewModel

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        //Don't create directly; ViewModel outlives Activity, so data won't survive
        //Not recommended
        //ViewModelProviders.of(<activity or fragment>).get(<viewmodel::class.java)
        viewModel = ViewModelProviders.of(this).get(MainViewModel::class.java)
        plusOneBtn.setOnClickListener {
            viewModel.counter++
            refreshCounter()
        }
        refreshCounter()
    }

    private fun refreshCounter() {
        infoText.text  = viewModel.counter.toString()
    }
}
```

#### 13.2.2 Passing Data to ViewModel

- Use ViewModelProvider.Factory

So data can be saved even after exiting the app

1. Modify ViewModel

```
class MainViewModel(countReserved: Int): ViewModel(){

    var counter = countReserved
}
```

1. Implement ViewModelProvider.Factory interface

```
class MainViewModelFactory(private val countReserved: Int) :ViewModelProvider.Factory {
    override fun <T : ViewModel?> create(modelClass: Class<T>): T {
        return MainViewModel(countReserved) as T
    }
}
```

1. Logic

```
class MainActivity : AppCompatActivity() {

    lateinit var viewModel: MainViewModel
    lateinit var sp: SharedPreferences

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)
        sp = getPreferences(Context.MODE_PRIVATE)
        val countReserved = sp.getInt("count_reserved",0)
        clearBtn.setOnClickListener {
            viewModel.counter = 0
            refreshCounter()
        }

        //Don't create directly; ViewModel outlives Activity, so data won't survive
        viewModel = ViewModelProviders.of(this,MainViewModelFactory(countReserved)).get(MainViewModel::class.java)
        plusOneBtn.setOnClickListener {
            viewModel.counter++
            refreshCounter()
        }
        refreshCounter()
    }

    private fun refreshCounter() {
        infoText.text  = viewModel.counter.toString()
    }

    override fun onPause() {
        super.onPause()
        sp.edit{
            putInt("count_reserved",viewModel.counter)
        }
    }
}
```

### 13.3 Lifecycles

- Always be aware of the Activity lifecycle so you can run logic at the right time. Example: if you start a network request in the UI and the UI is closed when the response arrives, you shouldn't continue processing the response.

- How to be aware of the Activity lifecycle from non-Activity classes: 1) embed a hidden Fragment in the Activity; 2) use a hand-written listener approach

- Hand-written listener approach — downside: lots of code in the Activity, not elegant

```
lateinit var observer: MyObserver

observer = MyObserver()

override fun onStart() {
        super.onStart()
        observer.activityStart()
    }

    override fun onStop() {
        super.onStop()
        observer.activityStop()
    }
```

- Lifecycles: lets any class easily be aware of the Activity lifecycle without writing lots of code in the Activity

1. Implement interface, write methods and annotations

```
class MyObserver : LifecycleObserver {

    private val TAG = "MyObserver"
    @OnLifecycleEvent(Lifecycle.Event.ON_START)
    fun activityStart() {
        Log.d(TAG, "activityStart: ")
    }

    @OnLifecycleEvent(Lifecycle.Event.ON_STOP)
    fun activityStop() {
        Log.d(TAG, "activityStop: ")
    }
}
```

- ON_CREATE, ON_START, and so on for other lifecycle events

- ON_ANY matches any lifecycle event

1. Activities and Fragments are LifecycleOwner instances

Call in Activity:

```
lifecycle.addObserver(MyObserver())
```

- Actively get lifecycle state — add constructor

```
class MyObserver(val lifecycle:Lifecycle) : LifecycleObserver {

    private val TAG = "MyObserver"
    @OnLifecycleEvent(Lifecycle.Event.ON_START)
    fun activityStart() {
        Log.d(TAG, "activityStart: ")
    }

    @OnLifecycleEvent(Lifecycle.Event.ON_STOP)
    fun activityStop() {
        Log.d(TAG, "activityStop: ")
    }
}
```

With the object you can call anywhere

Lifecycle.currentState to get current lifecycle state

### 13.4 LiveData

- Reactive programming component that holds data of any type and notifies observers when data changes. Usually used with ViewModel

- Wrap the counter with LiveData, then observe it in the Activity to get notified of changes

1. Modify MainViewModel

```
class MainViewModel(countReserved: Int): ViewModel(){

    //Generic Int; ML is MutableLiveData.
    //getValue reads LiveData
    //setValue on main thread only
    //postValue from background thread
    val counter = MutableLiveData<Int>()

    init {
        counter.value = countReserved
    }

    fun plusOne() {
        val count = counter.value ?: 0
        counter.value = count + 1
    }

    fun clear() {
        counter.value = 0
    }

}
```

1. Logic

```
class MainActivity : AppCompatActivity() {

    lateinit var viewModel: MainViewModel
    lateinit var sp: SharedPreferences

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        setContentView(R.layout.activity_main)

        lifecycle.addObserver(MyObserver(lifecycle))

        sp = getPreferences(Context.MODE_PRIVATE)
        val countReserved = sp.getInt("count_reserved",0)
        clearBtn.setOnClickListener {
            viewModel.clear()
        }

        //Don't create directly; ViewModel outlives Activity, so data won't survive
        viewModel = ViewModelProviders.of(this,MainViewModelFactory(countReserved)).get(MainViewModel::class.java)
        plusOneBtn.setOnClickListener {
            viewModel.plusOne()
        }
        //Any LiveData has observe()
        //First: LifecycleOwner; second: Observer callback on change
        viewModel.counter.observe(this, Observer { count ->
            infoText.text = count.toString()
        })
    }

    override fun onPause() {
        super.onPause()
        sp.edit{
            putInt("count_reserved",viewModel.counter.value ?: 0)
        }
    }
}
```

- lifecycle-livedata-ktx is designed for Kotlin and extends observe syntax. Type the version after observe

```
implementation "androidx.lifecycle:lifecycle-livedata-ktx:2.5.1"
```

```
//concise syntax
viewModel.counter.observe(this){ count ->
    infoText.text = count.toString()
}
```

Problem: exposing counter data externally breaks LiveData encapsulation

Solution: expose immutable LiveData externally so non-ViewModel code can only observe changes

```
class MainViewModel(countReserved: Int): ViewModel(){
    
    val counter:LiveData<Int>
        get() = _counter
    
    private val _counter = MutableLiveData<Int>()

    init {
        _counter.value = countReserved
    }

    fun plusOne() {
        val count = counter.value ?: 0
        _counter.value = count + 1
    }

    fun clear() {
        _counter.value = 0
    }

}
```

- Entity: corresponds to a database table

- Dao (Data Access Object): defines database operations

- Database: defines key database info and provides access instances

1. Entity class

```
//@Entity marks table entity
@Entity
data class User(var firstName:String,var lastName:String,var age: Int) {

    //id field: primary key, auto-generated
    @PrimaryKey(autoGenerate = true)

1. Dao — interface

```
//Room recognizes as Dao
@Dao
interface UserDao {
    
    //return generated primary key after insert
    //handle existing id on insert
    //on conflict: replace old row and continue transaction
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    fun insertUser(user: User):Long
    
    @Update
    fun updateUser(newUser: User)
    
    @Query("select * from User")
    fun loadAllUsers(): List<User>
    
    @Query("select * from User where age > :age")
    fun loadUserOlderThan(age: Int): List<User>
    
    @Delete
    fun deleteUser(user: User)
    
    @Query("delete from User where lastName = :lastName")
    fun deleteUserByLastName(lastName: String):Int
    
}
```

- Queries require SQL; parameters can be added to SQL statements

- For insert/update/delete with non-entity types, only @Query with SQL works — see last item

1. Database migration logic

```
//version and entities; comma-separate multiple entities
//extend RoomDatabase; provide abstract Dao accessors
@Database(version = 2, entities = [User::class,Book::class]) 1
abstract class AppDatabase : RoomDatabase() {
    abstract fun userDao():UserDao
    abstract fun bookDao(): BookDao 2

    companion object{
        
        //runs migration logic when upgrading v1 to v2
        val MIGRATION_1_2 = object : Migration(1, 2) {
            override fun migrate(database: SupportSQLiteDatabase) { 3
                database.execSQL("create table Book (id integer primary" +
                        " key autoincrement not null,name text not null," +
                        "pages integer not null)")
            }
        }
        
        //singleton
        private var instance: AppDatabase? = null

        @Synchronized
        fun getDatabase(context: Context): AppDatabase {
            instance?.let {
                return it
            }
            //first param must be applicationContext, not context
            //second param is AppDatabase class
            return Room.databaseBuilder(context.applicationContext,
            AppDatabase::class.java,"app_database")
                .allowMainThreadQueries()
                .addMigrations(MIGRATION_1_2) 4
                .build().apply {
                    instance = this
                }
        }

    }
}
```

### 13.6 WorkManager

Purpose: handles scheduled tasks. Automatically chooses AlarmManager or JobScheduler based on OS version. Supports periodic and chained tasks. Unrelated to Services — still runs after app exit and phone reboot. Suitable for periodic server sync tasks. Not necessarily on time; to save power, multiple tasks may be batched.

```
implementation "androidx.work:work-runtime:2.7.1"
```

1. Define a background task and implement logic

```
class SimpleWorker(context: Context, params:WorkerParameters):Worker(context,params) {
    override fun doWork(): Result {
    		//background work; not on main thread
        Log.d("SimpleWorker","do work in SimpleWorker")
        return Result.success()
        //Result.failure() means task failed
        //Result.retry() means retry later
    }
}
```

1. Configure run conditions and constraints, build work request

Add button; if button exceeds LL bounds use ScrollView to scroll

```
doWorkBtn.setOnClickListener {

    //build one-time work request
    val request = OneTimeWorkRequest.Builder(SimpleWorker::class.java).build()
    //PeriodWorkRequest for periodic work; min interval 15 minutes   
    WorkManager.getInstance(this).enqueue(request)
}
```

- With no constraints specified, runs immediately

#### 13.6.2 WorkManager for Complex Tasks

```
doWorkBtn.setOnClickListener {

    //build one-time work request
    //PeriodWorkRequest for periodic work; min interval 15 minutes
    val request = OneTimeWorkRequest.Builder(SimpleWorker::class.java)
        .setInitialDelay(5,TimeUnit.MINUTES)//delay before run; units: ms, s, m, h, d
        .addTag("simple")//tag for batch cancel
        //if doWork returns Result.retry()
        //combine with backoff to retry
        .setBackoffCriteria(BackoffPolicy.LINEAR,10,TimeUnit.SECONDS)
        //first param: LINEAR backoff on failure
        //EXPONENTIAL is exponential backoff
            //second/third: delay; min 10 minutes for periodic
        .build()
    
    WorkManager.getInstance(this).enqueue(request)
}

WorkManager.getInstance(this).cancelAllWorkByTag("simple")//cancel by tag
WorkManager.getInstance(this).cancelWorkById(request.id)//cancel by id; single only
WorkManager.getInstance(this).cancelAllWork()//cancel all work
//observe doWork result
WorkManager.getInstance(this).getWorkInfoByIdLiveData(request.id).observe(this){workInfo ->
    if ((workInfo.state == WorkInfo.State.SUCCEEDED)) {
        Log.d(TAG, "do work success")
    }else if(workInfo.state == WorkInfo.State.FAILED){
        Log.d(TAG, "do work failed")
    }
}
```

### 13.7 Kotlin: Building Domain-Specific Language (DSL) Syntax

- DSL (Domain-Specific Language): write code that looks like it breaks away from the original syntax structure.

- Implement DSL with higher-order functions. Similar to dependency library content

```
class Dependency{

    val libraries = ArrayList<String>()

    fun implementation(lib: String) {
        libraries.add(lib)
    }

}

fun dependencies(block: Dependency.() -> Unit): List<String> {
    val dependency = Dependency()
    dependency.block()
    return dependency.libraries
}

fun main() {
    //call
    dependencies {
        implementation("com.squareup.retrofit2:retrofit2:2.6.1")
        implementation("com.squareup.retrofit2:converter-gson2.6.1")
    }
    //get all dependencies added
    val libraries = dependencies {
        implementation("com.squareup.retrofit2:retrofit2:2.6.1")
        implementation("com.squareup.retrofit2:converter-gson2.6.1")
    }
    for (lib in libraries) {
        println(lib)
    }
}

- Dynamically generate HTML for tables in Kotlin

```
package com.android.jetpacktest

import java.lang.StringBuilder

class Td{
    var content = ""

    fun html() = "\n\t\t<td>$content</td>"
}

class Tr{
    private val children = ArrayList<Td>()

    fun td(block: Td.() -> String) {
        val td = Td()
        td.content = td.block()
        children.add(td)
    }

    fun html():String{
        val builder = StringBuilder()
        builder.append("\n\t<tr>")
        for (childTag in children) {
            builder.append(childTag.html())
        }
        return builder.toString()
    }
}

class Table{
    private val children = ArrayList<Tr>()

    fun tr(block: Tr.() -> Unit) {
        val tr = Tr()
        tr.block()
        children.add(tr)
    }

    fun html(): String {
        val builder = StringBuilder()
        builder.append("<table>")
        for (childTag in children) {
            builder.append(childTag.html())
        }
        builder.append("\n</table>")
        return builder.toString()
    }
}

fun table(block: Table.() -> Unit): String {
    val table = Table()
    table.block()
    return table.html()
}

fun main() {
    val html = table {
        tr{
            td { "Apple" }
            td { "Grape" }
            td { "Orange" }
        }
        tr {
            td { "Pear" }
            td { "Banana" }
            td { "Watermelon" }
        }
    }
    println(html)
}

fun main() {
    val html = table {
        repeat(2) {
            tr {
                val fruits = listOf("Apple","Grape","Orange")
                for (fruit in fruits) {
                    td { fruit }
                }
            }
        }
    }

    println(html)
}
```
