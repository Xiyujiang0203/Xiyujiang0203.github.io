3. Explain MVP and MVVM architectures
    <!-- image: removed (asset not available) -->

   MVP: The View holds a reference to the Presenter. When the user interacts with the View, data is sent to the Presenter. The Presenter holds a reference to the Model, forwards View data to the Model to fetch additional data, and when the Model returns data, the Presenter updates the View. The Presenter also holds a reference to the View.

   MVVM: The View holds a reference to the ViewModel and updates the UI by observing data. The ViewModel holds a reference to the Repository layer for business logic; the Repository is responsible for data fetching.

1. Why does ViewModel in MVVM preserve the UI state after screen rotation?

`ViewModelProvider` retrieves a ViewModel from `ViewModelStore`.
If retrieval fails, it creates a ViewModel via `ViewModelProvider.Factory`.

1. Explain the Handler mechanism and its use cases
   How it works:
    <!-- image: removed (asset not available) -->

    <!-- image: removed (asset not available) -->

Analogy: a child thread delivers messages to the main thread; the main thread checks its "mailbox" periodically.
Mechanism:

- When a thread needs Handler to process messages, it first creates a Looper and calls `prepare()` to create a message queue associated with the current thread.

- Then `loop()` starts the message loop, continuously reading messages from the queue.

- When a message is sent to the queue via `Handler.sendMessage()`, the Looper takes messages from the queue and passes them to the corresponding Handler.

- The Handler calls its own `handleMessage()` to process the message.
  Use cases: UI updates, background task handling, scheduled tasks, inter-thread communication.

6. How do Looper and Thread maintain a one-to-one relationship?

Via ThreadLocal: each thread's ThreadLocal stores at most one Looper; if one already exists it is reused, otherwise a new one is created. A thread can have multiple Handlers sharing the same Looper.

7. Glide's three-level cache? What are the three levels?
Active cache — images visible on screen; cleared when the page is destroyed
Memory cache — cleared when the app closes
Disk cache — stored on the device
Network cache (not counted as a cache level)

8. OkHttp interceptors?
Interceptors use the Chain of Responsibility pattern: an abstract interface, concrete roles, then assembly.

- RetryAndFollowUpInterceptor: retry interceptor — decides whether to retry; returns error if max retries exceeded, otherwise retries.

- BridgeInterceptor: request interceptor — adds common request headers

- CacheInterceptor: cache interceptor — checks for cached responses

- ConnectInterceptor: connection interceptor — retrieves data from existing long connections

- Call execution interceptor

- Callback to notify the business layer of results

- Custom interceptors
