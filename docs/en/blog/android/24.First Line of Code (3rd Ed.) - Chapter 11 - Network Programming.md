```
                connection?.disconnect()
```

### 11.5 Network Request Callback Patterns

1. HTTP requests must run on a child thread, otherwise they block the main thread. Use callbacks to handle returned data, otherwise you won't receive it

- Network request utility class

1. HttpURLConnection

```
object HttpUtil {

    fun sendHttpRequest(address:String,listener: HttpCallbackListener){
        var connection : HttpURLConnection? = null

            try {
                var response = StringBuilder()
                val url = URL(address)
                connection = url.openConnection() as HttpURLConnection
                connection!!.connectTimeout = 8000
                connection!!.readTimeout = 8000
                val input = connection!!.inputStream
                //read the input stream below
                val reader = BufferedReader(InputStreamReader(input))
                reader.use{
                    reader.forEachLine {
                        response.append(it)
                    }
                }
                listener.onFinish(response.toString())
            } catch (e: Exception) {
                e.printStackTrace()
                listener.onError(e)
            }finally {
                connection?.disconnect()
            }
    }

}
```

### 11.6 Retrofit

OkHttp focuses on low-level communication; Retrofit focuses on high-level API encapsulation

Project homepage: https://github.com/square/retrofit

#### 11.6.1 Basic Usage

- Configure a base URL so you only need relative paths when specifying server addresses — no need to pass full URLs every time

- Categorize server APIs and define APIs of the same type in the same interface file

- No need to handle network communication details — just declare methods and return types in the interface file

- Use annotations to specify which server API each method maps to and what parameters to provide

- When you call the method, Retrofit automatically sends a request to the server API and parses the response into the declared return type

1. Add dependencies: include okhttp, gson, and retrofit together

```
implementation 'com.squareup.retrofit2:retrofit:2.6.1'
implementation 'com.squareup.retrofit2:converter-gson:2.6.1'
```

1. Retrofit converts JSON to objects, so add an App class

1. Categorize by server API functionality, create different interface files, and define specific server API methods in each

If the server has only one JSON endpoint, you only need one interface file with one method

```
/**
 * Naming: <Feature>Service

interface AppService {

    /**
     * @GET: Retrofit sends GET; path is relative; base URL set at build time
     * Return Call<T> with generic type for parsed response.
     * JSON array of App -> List<App>
```

#### 11.6.3 Best Practices for the Retrofit Builder

- Ordinary method to get interface dynamic proxy

```
val retrofit = Retrofit.Builder()
    .baseUrl("http://10.0.2.2")
    .addConverterFactory(GsonConverterFactory.create())
    .build()
val appService = retrofit.create(AppService::class.java)
```

- Retrofit object is globally shared; only the Class type differs — optimize by encapsulating the first part

```
//object makes ServiceCreator singleton
object ServiceCreator {
    
    //base URL
    private const val BASE_URL ="http://10.0.2.2/"

    //private: internal only
    private val retrofit = Retrofit.Builder()
        .baseUrl(BASE_URL)
        .addConverterFactory(GsonConverterFactory.create())
        .build()
    
    //public create(Class): full proxy creation
    fun <T> create(serviceClass: Class<T>): T = retrofit.create(serviceClass)
    
}
```

- Usage

```
val appService = ServiceCreator.create(AppService::class.java)
//then call any interface method
```

- Further optimization using generic reification from Chapter 10

```
//add to singleton above
		/**
     * reified requires inline and no type param on function
     * reified generic type
     * calls create(Class) above
     */
    inline fun<reified T> create(): T = create(T::class.java)
```

- Usage after adding


### 11.7 Kotlin Classroom: Coroutines — Writing Highly Concurrent Programs

- Coroutines are similar to threads

- A coroutine is a lightweight thread; switching between coroutines is implemented at the language level

- Threads are heavyweight and rely on OS scheduling for switching

- Simulate multithreaded programming on a single thread; suspend and resume are entirely decided by the language

- Starting 100,000 threads is impossible; 100,000 coroutines might be feasible

#### 11.7.1 Basic Coroutine Usage

#### 11.7.2 More Scope Builders

Problem: if a network request in a coroutine hasn't finished when the Activity is closed, the coroutine should exit too. For top-level coroutines you must call methods to close them yourself — maintenance cost is too high. Example:

```
fun main() {
    val job = GlobalScope.launch { 
        //business logic
    }
    job.cancel()
}
//uncommon in real projects
```

- More common usage

```
fun main() {
    val job = Job()
    val scope = CoroutineScope(job)
    scope.launch { 
        //business logic
    }
    job.cancel()
    //all launches tied to job; one cancel stops all
}
```

Note: CoroutineScope is more suitable for real projects; runBlocking is convenient for testing

- async function: create a coroutine and get a return value. Note: only callable in coroutine scopes. To get the result, call await on the returned Deferred object

```
fun main() {
    runBlocking {
        val result = async {
            5 + 5
        }.await()
        println(result)
    }
}
```

- Problem

```
fun main() {
    runBlocking {
        val start = System.currentTimeMillis()
        //await blocks until async completes
        val result = async {
            delay(1000)
            5 + 5
        }.await()
        val result2  = async {
            delay(1000)
            4+6
        }.await()
        println("result is ${result + result2}")
        val end = System.currentTimeMillis()
        println("cost ${end - start}ms.")
    }
}
Takes 2+ seconds
```

- Solution

```
fun main() {
		//call await when value needed
    runBlocking {
        val start = System.currentTimeMillis()
        //await blocks until async completes
        val deferred1 = async {
            delay(1000)
            5 + 5
        }
        val deferred2  = async {
            delay(1000)
            4+6
        }
        println("result is ${deferred1.await()+deferred2.await()}")
        val end = System.currentTimeMillis()
        println("cost ${end - start}ms.")
    }
}
//~1s+
```

- withContext function: a suspend function, can be thought of as a simplified async

```
fun main() {
    runBlocking {
        //runs block immediately, blocks thread, returns last line
        //equivalent to async{5+5}.await()
        //difference: forced dispatcher param
        //network on main-thread coroutine still errors
        //dispatcher selects thread pool
        val result = withContext(Dispatchers.Default){
            5 + 5
        }
    }
}
```

- Dispatchers.Default — uses a low-concurrency thread strategy by default; for CPU-intensive tasks, too much concurrency can hurt efficiency

- Dispatchers.IO — uses a high-concurrency strategy

- Dispatchers.Main — doesn't start a child thread; runs on the main thread. Only usable in Android projects; using this in pure Kotlin programs will fail

Note: besides coroutineScope, other scope builders can specify a thread; withContext requires it

- suspendCoroutine function takes a Lambda parameter, immediately suspends the current coroutine, then runs the Lambda code on an ordinary thread

```
    suspend fun request(address:String): String{
        //suspendCoroutine in coroutine or suspend fn
        //suspend coroutine; run lambda on thread
        //resume/resumeWithException continues coroutine
        //value passed becomes suspendCoroutine return
        return suspendCoroutine { continuation ->
            HttpUtil.sendHttpRequest(address,object:HttpCallbackListener{
                override fun onFinish(response: String) {
                    continuation.resume(response)
                }

                override fun onError(e: Exception) {
                    continuation.resumeWithException(e)
                }
            })
        }
    }

    //see ch.15 for architecture
```

```
suspend fun <T> Call<T>.await():T{
    return suspendCoroutine { continuation ->  
        enqueue(object : Callback<T>{
            override fun onResponse(call: Call<T>, response: Response<T>) {
                val body  = response.body()
                if (body != null) {
                    continuation.resume(body)
                }else{
                    continuation.resumeWithException(RuntimeException("response body is null"))
                }
            }
```
